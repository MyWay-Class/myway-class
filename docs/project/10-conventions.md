# 컨벤션

## 목적
코드베이스와 문서를 일관되게 유지하고, AI 에이전트가 프로젝트 스타일을 바로 따를 수 있게 한다.

## 범위
- TypeScript 코드 스타일
- 명명 규칙과 파일 배치
- API 응답 형식
- 로그와 진단 정보
- 문서, 프롬프트, ADR

## 입력
- 스택 결정
- 모듈 구조
- TypeScript 공식 문서의 스타일 가이드

## 출력
- 표준 개발 규칙
- TypeScript 코드 스타일 가이드
- AI가 생성한 코드에 대한 적용 규칙

## 규칙
- 애플리케이션 코드는 모두 TypeScript로 작성한다.
- JavaScript는 앱 코드에서 사용하지 않는다.
- JS는 도구 설정, 빌드 설정, 외부 도구 호환이 꼭 필요한 경우에만 제한적으로 허용한다.
- API 응답은 공통 봉투를 유지한다.
- 프롬프트와 아키텍처 결정은 버전 관리한다.
- 파일 이름은 기능 중심으로 짓는다.
- 기교보다 단순하고 명확한 코드를 우선한다.

## TypeScript 스타일 가이드

### 1. 기본은 타입 추론을 우선한다
- 로컬 변수는 타입이 명확하면 TypeScript의 추론을 그대로 사용한다.
- 가독성, 공개 API 명확성, 안전성에 도움이 될 때만 타입을 명시한다.
- 가능하다고 해서 모든 곳에 주석처럼 타입을 붙이지 않는다.
- `string`, `number`, `boolean`, `bigint`, `symbol`, `null`, `undefined` 같은 기본형은 공식 타입 체계를 그대로 사용한다.

### 2. `any` 대신 `unknown`을 사용한다
- 문서화된 예외가 없는 한 애플리케이션 코드에서 `any`는 금지한다.
- 신뢰할 수 없는 외부 입력은 `unknown`으로 받고, 사용하기 전에 좁힌다.
- 알 수 없는 입력은 로직 깊은 곳이 아니라 경계에서 변환한다.

### 3. 사용 전에 좁힌다
- JSON, query param, storage, 외부 API 값은 검증한다.
- type guard, `typeof`, `instanceof`, discriminated union, schema validation을 활용한다.
- 파싱한 객체가 기대한 필드를 가진다고 가정하지 않는다.

### 4. 객체 형태는 `interface`, 합집합과 계산 타입은 `type`을 우선한다
- DTO, 설정 객체, 재사용 가능한 객체 계약에는 `interface`를 사용한다.
- 합집합, 교집합, 매핑 타입, 유틸 성격 조합에는 `type`을 사용한다.
- 나중에 확장될 가능성이 있으면 `interface`를 우선한다.
- 고정 길이의 서로 다른 값 묶음만 tuple로 표현한다.
- 객체의 필수/선택 속성은 공식 문서의 객체 타입 규칙에 맞춰 명시한다.

### 5. `enum`은 아껴 쓴다
- 대부분의 도메인 상태에는 문자열 리터럴 합집합을 우선한다.
- 닫힌 런타임 값 집합이 가독성이나 통합에 정말 도움이 될 때만 `enum`을 사용한다.
- 상수 집합을 런타임으로 만들 필요가 없으면 `as const` 객체와 합집합 타입을 우선한다.

### 6. null 가능성은 명시한다
- 필드가 진짜로 비어 있을 수 있으면 `null`, 선택값이면 `undefined`를 사용한다.
- 둘을 이유 없이 섞지 않는다.
- 공개 API 응답은 프로퍼티가 생략될 수 있는지, `null`인지, 둘 다 가능한지 문서로 남긴다.

### 7. 함수는 작게 유지하고 경계에서 타입을 명시한다
- 입력 검증은 라우트와 서비스 경계에서 처리한다.
- 내부 헬퍼는 이미 검증된 값을 받는다.
- 반환 타입은 가능한 한 분명해야 하며, 공개 계약이면 명시한다.

### 8. 상태가 있는 흐름에는 discriminated union을 우선한다
- 값이 여러 상태를 가질 수 있으면 공통 `status` 또는 `kind` 필드를 둔다.
- AI 작업, 미디어 파이프라인 상태, 비동기 흐름에 특히 적합하다.
- 상태 분기는 `switch`와 좁히기를 우선하고, 분기 누락이 생기지 않게 한다.

### 9. 변경 의도가 없으면 `readonly`를 사용한다
- 설정, 프롬프트, 요청 스냅샷은 불변 흐름을 우선한다.
- 공유 상태의 우발적 변경을 막는다.

### 10. 비동기 코드는 직접적으로 쓴다
- 가독성을 위해 `async/await`를 사용한다.
- 콜백 중첩과 불필요한 Promise 체이닝은 피한다.
- 에러는 의미 있게 복구하거나 변환할 수 있는 계층에서 처리한다.

### 11. 공개 계약은 명시적으로 내보낸다
- 공유 모듈에서 interface, type, service contract를 export 한다.
- 다른 모듈이 정말 필요로 하지 않으면 private helper는 export하지 않는다.
- 공개 타입에 구현 상세를 누설하지 않는다.

### 12. 주석은 동작이 아니라 의도에 쓴다
- 이유가 코드만 봐서는 분명하지 않을 때만 주석을 단다.
- 주석은 tradeoff, 제약, 특이 동작을 설명해야 한다.
- 코드가 이미 명확히 말하는 내용을 다시 적지 않는다.

## 공식 문서 기준
- TypeScript 공식 문서의 기본형, 객체 타입, union, narrowing, strict null checks 관점을 따른다.
- JavaScript 공식 문서는 언어 동작 확인용으로만 참고하고, 앱 코드 스타일 기준으로는 채택하지 않는다.
- 이 프로젝트의 구현 코드는 TypeScript 중심이며, JavaScript 전용 컨벤션 문서는 만들지 않는다.
- 공식 문서 참고가 필요하면 먼저 TypeScript 핸드북을 보고, 그다음 MDN JavaScript 문서를 참고한다.

## 명명 규칙
- 파일 이름은 기능 중심으로 짓는다.
- 모듈에는 명확한 명사를 쓰고, handler와 action에는 명확한 동사를 쓴다.
- 라우트 이름은 구현보다 도메인에 맞춘다.

## 파일 배치
- 애플리케이션 코드는 `frontend/`와 `backend/`에 둔다.
- 브라우저 UI 코드는 `frontend/`에 둔다.
- API, Workers, 서비스 코드는 `backend/`에 둔다.
- 공통 계약은 `packages/shared`에 둔다.
- 설계 문서는 `docs/`에 둔다.
- AI 협업 규칙은 `docs/ai-context`에 둔다.

## API 형식
- 모든 응답은 공통 봉투를 사용한다.
- 에러 응답은 짧고 실행 가능해야 한다.
- AI가 만든 구조화 출력은 클라이언트로 보내기 전에 정규화한다.

## 로깅
- 디버깅, 추적, 비용 분석에 도움이 되는 정보만 기록한다.
- 비밀값, API 키, 민감한 원문 payload는 기록하지 않는다.
- AI 호출에는 공급자, 지연 시간, 성공 여부, 토큰 관련 메타데이터를 남긴다.

## 문서와 프롬프트
- 프롬프트는 버전이 있는 markdown 파일에 둔다.
- 아키텍처 결정은 ADR에 남긴다.
- 프롬프트가 동작을 바꾸면 관련 설계 문서를 함께 갱신한다.

## 예외 상황
- 데모 전용 코드는 반드시 표시한다.
- 컨벤션 변경은 문서화된 이유가 있을 때만 허용한다.
- 서드파티 타입은 경계에서 좁고 명시적인 예외가 필요할 수 있다.

## 실패 모드
- 일관성 없는 명명
- 추적되지 않은 프롬프트 변경
- 공유 코드의 숨은 결합
- `any` 과다 사용
- 과도한 타입 명시로 읽기 어려워지는 경우

## 검증 기준
- 새 기여자가 프로젝트 스타일에 맞는 코드를 작성할 수 있다.
- AI 에이전트가 추가 맥락 없이도 규칙을 사용할 수 있다.
- 스타일 가이드가 추론, `unknown`, 안전한 타이핑에 대한 TypeScript 문서와 맞는다.
- JavaScript 전용 스타일 규칙이 앱 코드 규칙으로 섞여 있지 않다.
