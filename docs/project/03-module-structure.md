# 모듈 구조

## 목적
각 기능이 어디에 위치해야 하는지 정해 구현이 깔끔하게 나눌 수 있게 한다.

## 역할
- 이 문서는 구현 기준의 상세 모듈 문서다.
- 요약 허브는 `docs/context/module-structure.md`다.
- context 문서와 같은 내용을 반복하지 않고, 실제 배치 기준을 적는다.

## 범위
- 웹앱 모듈
- API 라우트
- 공통 타입
- AI 서비스

## 입력
- 계획된 스택
- 제품 비전에서 정리된 기능 목록

## 출력
- 파일과 폴더의 책임 분리
- 의존성 규칙

## 규칙
- 라우트는 조정만 하고 실제 일은 서비스가 한다.
- UI 컴포넌트는 저장소 내부 구조를 알지 못한다.
- 공통 타입은 `packages/shared`에 둔다.
- 프롬프트 로직은 비즈니스 로직과 분리한다.

## 권장 구조
- `frontend/src/components`
- `frontend/src/pages`
- `frontend/src/lib`
- `backend/src/routes`
- `backend/src/services`
- `backend/src/middleware`
- `backend/src/types`
- `packages/shared/src`
- `docs/`

## 도메인 책임
- 인증: 세션, 로그인, 역할
- 강의: 강의 콘텐츠와 강의 페이지
- AI: RAG, 퀴즈, 요약, 의미 검색
- 미디어: STT, 타임스탬프, 오디오 추출
- 숏폼: 생성, 편집, 공개
- 커뮤니티: 반응, 랭킹, 모더레이션

## 예외 상황
- 한 기능이 여러 도메인을 넘나들면 단일 조정 서비스를 둔다.
- 데모 전용 코드는 운영 코드 경로와 분리한다.

## 실패 모드
- 너무 비대한 라우트 핸들러
- 순환 import
- 여러 계층에 중복된 기능

## 검증 기준
- 새 구현 작업을 정확히 하나의 주 모듈에 배치할 수 있다.
- 모듈 트리가 모호함 없이 확장 가능하다.
