# 기술 스택

## 목적
이후 문서와 코드가 같은 전제를 공유하도록 구현 스택을 확정한다.

## 범위
- 프론트엔드 실행 환경과 도구
- 백엔드 실행 환경과 엣지 플랫폼
- 데이터베이스와 객체 저장소
- AI 공급자와 fallback 전략

## 입력
- 빠른 개발이 필요한 상황
- 데모에서 빠른 응답이 필요한 상황
- AI 비중이 큰 기능 흐름

## 출력
- 최종 스택 결정
- 스택을 선택한 이유
- 의도적으로 제외한 범위

## 규칙
- 프론트엔드는 `React + TypeScript + Vite`를 사용한다.
- 백엔드는 Cloudflare Workers 위의 `Hono + TypeScript`를 사용한다.
- 메타데이터용 주 DB는 `Cloudflare D1`을 사용한다.
- 큰 미디어 자산은 `Cloudflare R2`에 저장한다.
- 공개 호스팅은 `Cloudflare Pages`를 사용한다.
- 배포와 실행은 Cloudflare 생태계를 기준으로 한다.
- AI는 `Cloudflare AI(Workers AI)`, `Gemini 무료 API`, `Codex 5.2`, `Ollama 3.2` 순으로 우선순위를 둔다.

## 권장 스택
- 웹: React, TypeScript, Vite, Pages
- API: Hono, TypeScript, Workers
- DB: D1
- 저장소: R2
- 캐시/세션: Workers 캐시 또는 KV
- AI: Cloudflare AI → Gemini 무료 API → Codex 5.2 → Ollama 3.2

## 선택 이유
- 하나의 언어 계열로 개발 부담을 줄일 수 있다.
- 엣지 호스팅은 cold start 문제를 줄여준다.
- D1과 R2는 메타데이터와 자산을 분리하기에 잘 맞는다.
- Cloudflare 배포는 프론트, 백엔드, 저장소를 같은 운영 모델로 묶기 쉽다.
- 지정한 AI 공급자 조합은 무료/로컬/외부 API를 단계적으로 묶기 좋다.

## 예외 상황
- AI 작업이 너무 비싸면 더 작은 프롬프트로 쪼갠다.
- 내용이 크면 D1에는 메타데이터만 두고 본문은 R2에 저장한다.
- 기능이 데모 전용이면 문서에 반드시 표시한다.
- 공급자는 항상 우선순위에 따라 선택하고, 실패하면 다음 공급자로 넘긴다.

## 실패 모드
- D1을 대용량 파일 저장소처럼 쓰는 경우
- 명확한 이점 없이 여러 백엔드 스택을 도입하는 경우
- AI 공급자 전략이 문서와 다르게 흘러가는 경우

## 검증 기준
- 선택한 스택이 다른 문서와 일관된다.
- 다른 AI가 이 문서만 읽어도 런타임과 저장 모델을 추론할 수 있다.
