# 파일 분리 규칙

## 목적
하나의 파일이 너무 비대해지지 않도록 분리 기준을 정한다.

## 기준
- 한 파일은 한 책임만 가진다.
- 한 파일이 여러 역할을 하기 시작하면 분리한다.
- 공통되는 부분과 도메인별 부분이 섞이면 나눈다.
- 200줄을 넘기기 시작하면 분리 가능성을 본다.
- 300줄을 넘기면 반드시 분리한다.
- `types`, `constants`, `config` 파일은 예외적으로 더 길어질 수 있지만, 설명용 로직이 붙기 시작하면 다시 나눈다.

## 반드시 분리하는 경우
- 역할이 2개 이상이다.
- 일부 로직을 다른 파일에서 재사용할 수 있다.
- 수정 시 영향 범위가 크다.
- 테스트가 파일 하나에 과하게 몰린다.
- 조건 분기나 중첩이 늘어나 읽기 어려워진다.
- 함수가 계속 늘어나거나 비슷한 로직이 반복된다.

## 분리 우선순위
- `App`이나 `Page`는 상태 조합과 화면 조립만 남긴다.
- 라우트나 컨트롤러는 요청 파싱과 응답 반환만 남긴다.
- 비즈니스 규칙은 `service`나 `shared`의 도메인 함수로 옮긴다.
- 반복되는 계산은 `helper`로 묶는다.
- UI는 섹션 단위 컴포넌트로 나눈다.

## 분리 신호
- 같은 종류의 함수가 계속 추가된다.
- 조건 분기가 많아져 읽기 어려워진다.
- 테스트가 파일 하나에 집중되어 커진다.
- 수정할 때마다 다른 기능이 같이 흔들린다.

## 분리 예시
- UI 컴포넌트가 길어지면 `components/`와 `pages/`로 나눈다.
- API 라우트가 비대해지면 `routes/`와 `services/`로 나눈다.
- 공통 타입이 반복되면 `packages/shared/`로 옮긴다.
- 문서가 길어지면 `README`와 companion doc로 나눈다.

## 금지
- 이유 없이 파일만 잘게 쪼개지 않는다.
- 같은 책임을 가진 파일을 여러 개로 흩뿌리지 않는다.
- 분리한 뒤에는 파일명이 책임을 드러내야 한다.

## 검증 기준
- 파일 하나를 열었을 때 역할이 즉시 보인다.
- 비슷한 책임은 같은 층에 모여 있다.
