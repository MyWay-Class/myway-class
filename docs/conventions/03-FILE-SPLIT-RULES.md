# 파일 분리 규칙

## 목적
하나의 파일이 너무 비대해지지 않도록 분리 기준을 정한다.

## 기준
- 한 파일은 한 책임만 가진다.
- 한 파일이 여러 역할을 하기 시작하면 분리한다.
- 공통되는 부분과 도메인별 부분이 섞이면 나눈다.
- 200줄을 넘기기 시작하면 분리 가능성을 본다.

## 분리 신호
- 같은 종류의 함수가 계속 추가된다.
- 조건 분기가 많아져 읽기 어려워진다.
- 테스트가 파일 하나에 집중되어 커진다.
- 수정할 때마다 다른 기능이 같이 흔들린다.

## 분리 예시
- UI 컴포넌트가 길어지면 `components/`와 `pages/`로 나눈다.
- API 라우트가 비대해지면 `routes/`와 `services/`로 나눈다.
- 공통 타입이 반복되면 `packages/shared/`로 옮긴다.
- 문서가 길어지면 `README`와 companion doc로 나눈다.

## 금지
- 이유 없이 파일만 잘게 쪼개지 않는다.
- 같은 책임을 가진 파일을 여러 개로 흩뿌리지 않는다.

## 검증 기준
- 파일 하나를 열었을 때 역할이 즉시 보인다.
- 비슷한 책임은 같은 층에 모여 있다.
