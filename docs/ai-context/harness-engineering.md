# 하네스 엔지니어링 가이드

## 목적
MyWayClass에서 AI와 협업할 때, 컨텍스트 과부하를 줄이고 리포지토리를 단일 진실 원천으로 사용하기 위한 규칙을 정리한다.

## 범위
- AI 에이전트가 읽는 핵심 가이드의 크기와 구조
- PRD, 작업 계획, 의사결정 기록 방식
- 기능 문서, 보조 문서, appendix 분리 기준
- 작업이 끝난 뒤 컨텍스트를 정리하는 방법

## 핵심 원칙
- 리포지토리에 남지 않은 정보는 작업 기준으로 삼지 않는다.
- 프로젝트의 설명, 결정 이유, 우선순위는 문서로 남긴다.
- AI에게 주는 핵심 가이드는 200줄 내외로 유지한다.
- 핵심 가이드가 길어지면 보조 문서로 분리한다.
- 한 작업이 끝나면 그 작업의 임시 맥락은 잊어버리고, 필요한 내용만 문서에 남긴다.

## 문서화 규칙
- 모든 중요한 결정에는 `왜(Why)`와 `왜 이 선택을 했는지(Decision)`를 남긴다.
- 기능 우선순위는 `Must / Should / Could / Won't`로 구분한다.
- 새 기능은 바로 구현하지 말고, 먼저 요구사항과 검증 기준을 짧게 적는다.
- 큰 기능은 다음 순서로 쪼갠다.
  1. 현재 맥락 확보
  2. 요구사항 정의
  3. UX 또는 흐름 정리
  4. 의존성 확인
  5. 구현 계획 작성
  6. 리스크 정리
  7. 테스트 작성

## 작업 방식
- 회의 내용, 채팅 내용, 구현 메모는 필요한 부분만 문서로 승격한다.
- 반복해서 설명해야 하는 정보는 문서가 부족하다는 신호로 본다.
- 같은 내용을 계속 다시 설명해야 하면 문서를 더 잘게 쪼갠다.
- AI에게는 완성된 문서와 필요한 appendix만 주고, 과도한 원문 컨텍스트는 주지 않는다.

## MyWayClass 적용 방식
- 메인 진입점은 `agent.md`다.
- 200줄을 넘는 설명은 `docs/ai-context/`의 보조 문서로 옮긴다.
- 기능별 계획은 `docs/templates/`와 `docs/dev-logs/`를 함께 사용한다.
- 구조와 경계는 `docs/context/`와 `docs/structure/`에 둔다.
- 작업 우선순위는 `docs/conventions/01-MOSCOW-PRIORITY.md`를 따른다.

## 실패 모드
- 핵심 규칙이 너무 길어져서 AI가 시작도 전에 맥락을 소모하는 상태
- 결정 이유가 문서에 남지 않아 나중에 의도가 사라지는 상태
- 같은 내용을 여러 채널에 흩어 적어서 기준이 충돌하는 상태
- PRD 없이 기능을 바로 구현해 괴물 같은 코드가 생기는 상태

## 검증 기준
- 새 AI가 이 문서만 읽어도 작업 원칙을 이해할 수 있다.
- 핵심 가이드와 보조 문서의 경계가 분명하다.
- 중요한 결정이 리포지토리 안에 남아 있다.
