# AI 에이전트 규칙

## 목적
다른 AI가 이 저장소에서 작업할 때의 특수 규칙만 적는다.
기본 작업 원칙은 루트 `agent.md`를 따른다.

## 문서 유형
보조 원본 문서입니다. 루트 `agent.md`의 특수 규칙과 하네스 기준을 담습니다.

## 역할
- 이 문서는 AI 협업 보조 문서다.
- 루트 `agent.md`의 복제본이 아니다.
- 컨텍스트를 줄이기 위한 하네스 규칙과 문서 승격 기준만 다룬다.

## 읽는 순서
1. 루트 `agent.md`
2. `docs/ai-context/harness-engineering.md`
3. `docs/ai-context/ai-engine-provider-connection.md`
4. 필요한 경우 `docs/dev-logs/`와 관련 문서

## 특화 규칙
- 이 문서는 루트 규칙을 반복하지 않는다.
- 다른 AI에게 넘길 때는 작업 목적, 참조 문서, 완료 기준만 짧게 남긴다.
- 새로 얻은 정보는 대화에 남기지 말고, 문서나 `docs/dev-logs/`에 승격한다.
- 과도한 원문 컨텍스트는 주지 말고, 필요한 링크만 남긴다.

## LLM 사용 방식
- 작업 지시는 `목표 / 대상 / 완료 기준 / 참조 문서` 형식으로 짧게 쓴다.
- 컨벤션은 별도 문서에 두고, 지시에서는 파일 경로만 참조한다.
- 분석 요청에는 파일명과 라인 번호를 함께 적는다.
- 결과를 채택할 때는 왜 그 답을 골랐는지 `docs/dev-logs/`에 기록한다.

## 작업 지시 템플릿
```text
[GOAL]
무엇을 만들거나 바꿀지 한 줄로 적는다.

[INPUT]
- 현재 전제
- 이미 존재하는 파일, 엔티티, API, 규칙

[SCOPE]
- 이번 작업에서 건드릴 범위
- 건드리지 않을 범위

[TASK]
1. 실제 구현할 일 1
2. 실제 구현할 일 2
3. 실제 구현할 일 3

[CONSTRAINTS]
- 반드시 지켜야 하는 규칙
- 예외 처리, 응답 형식, 구조 분리 기준

[OUTPUT]
- 최종 산출물
- 생성/수정될 파일 종류

[ACCEPTANCE CRITERIA]
- 완료 여부를 판단할 기준
- 동작 확인 조건
```

## 다답안 비교
- 루트 `agent.md`의 비교 기준을 따른다.
- 필요한 경우만 비교하되, 선택 이유는 `docs/dev-logs/`에 남긴다.

## 하네스 원칙
- 핵심 기준은 한 곳에만 둔다.
- 보조 문서는 역할이 다를 때만 유지한다.
- 문서 경계가 흐려지면 먼저 요약 문서와 상세 문서를 분리한다.
- 루트 `agent.md`와 충돌하면 루트와 프로젝트 상세 문서를 우선한다.
- 브랜치 최신화는 `git pull --rebase` 또는 `git fetch` 후 `git rebase`로 한다.
- merge 커밋이 꼭 필요하면 작업 이유를 `docs/dev-logs/`에 남긴다.

## 실패 모드
- 루트 규칙과 내용이 거의 같은 보조 문서가 늘어나는 상태
- AI가 어디를 우선 읽어야 하는지 헷갈리는 상태
- 대화 메모가 문서보다 더 중요해지는 상태
- pull 과정에서 불필요한 merge commit이 쌓이는 상태
